# Upstream Runtime Integration

## Goal

This document captures the recommended built-in integration strategy for these upstream projects:

- `openclaw`
- `zeroclaw`
- `ironclaw`
- `codex`
- `sdkwork-api-router`

The target product is the current `claw-studio` workspace, with desktop built-in integration as the first priority and API Router as the shared control-plane foundation.

## Current Claw Studio Baseline

The desktop host already has the right native backbone:

- a Rust/Tauri command that installs software from a registry and emits progress events
- install control levels modeled as `managed`, `partial`, and `opaque`
- runtime platform routing for `windows`, `macos`, `ubuntu`, `wsl`, and container-aware flows
- an API Router installer service that already writes client configuration for Codex, Claude Code, OpenCode, OpenClaw, and Gemini

The most important local gaps are:

- `packages/sdkwork-claw-apirouter/src/services/apiRouterService.ts` is still mock-backed
- `packages/sdkwork-claw-extensions/src/services/extensionService.ts` is still mock-backed
- the vendored Hub Installer registry previously treated `zeroclaw` and `ironclaw` as `pnpm` tools and pointed at stale fork URLs

## Integration Principles

- Keep `@sdkwork/claw-web` and `@sdkwork/claw-desktop` thin. Runtime management belongs in shared infrastructure and the Tauri Rust host, not in the web entry package.
- Use Rust as the parent supervisor whenever the upstream ships a real native runtime.
- Prefer upstream-supported integration surfaces over reverse-engineered wrappers.
- Treat installation, process supervision, config writing, migration, and upgrades as separate layers.
- Preserve compatibility by detecting capabilities and install provenance, not only version numbers.

## Upstream Recommendations

## OpenClaw

`openclaw` is Node/TypeScript first. Its package entrypoint is `openclaw.mjs`, it exports a large plugin SDK surface, and its update flow is CLI-driven with `stable`, `beta`, `dev`, `doctor`, and JSON output.

Recommended strategy:

- install and supervise it as an external managed runtime
- let Rust own Node version selection, install location, process lifecycle, logs, and config wiring
- do not attempt deep Rust in-process embedding
- prefer file, CLI, or HTTP bridges over internal package coupling

## ZeroClaw

`zeroclaw` is Rust first. It exposes a native binary, a trait-driven runtime model, and explicit OpenClaw migration support. Its official installer prefers prebuilt binaries where possible and falls back to source build or Docker mode.

Recommended strategy:

- treat ZeroClaw as a Rust-native companion runtime
- prefer official prebuilt release assets where platform support is mature
- use source install as the reliable managed fallback, especially on Windows
- keep migration flows aligned with upstream `migrate openclaw`

## IronClaw

`ironclaw` is also Rust first, but its runtime is heavier and includes a WASM plus MCP extension architecture, richer import flows, and a release pipeline generated by `cargo-dist`.

Recommended strategy:

- treat IronClaw as a Rust-native companion runtime
- prefer official release installers or archives for end users
- use source install for managed development and controlled fallback
- preserve its extension model as an external runtime concern rather than flattening it into Claw Studio packages

## Codex

The maintained Codex implementation is Rust. The JavaScript CLI entrypoint is a thin launcher for platform-specific native binaries. The strongest supported integration surface is `codex app-server`, which exposes a JSON-RPC protocol over stdio or experimental websocket.

Recommended strategy:

- install Codex from official native distribution channels or source build for development
- integrate interactively through `codex app-server` over stdio
- model Codex as a protocol-driven tool runtime, not as a text-only CLI

## SDKWork API Router

`sdkwork-api-router` is the correct local control plane and provider data plane. It already documents standalone server mode and desktop-oriented embedded mode, loopback trust boundaries, SQLite for local persistence, and OS keyring backed secrets.

Recommended strategy:

- make API Router the shared local control plane for provider routing and credentials
- replace the current mock-only `sdkwork-claw-apirouter` frontend service with the real backend
- reuse it for desktop built-in routing before adding more client-side provider logic

## Target Architecture

The recommended desktop-first architecture is:

1. Tauri and Rust host
2. Hub Installer registry and install records
3. process supervisor and protocol bridges
4. embedded `sdkwork-api-router`
5. external managed runtimes for OpenClaw, ZeroClaw, IronClaw, and Codex
6. thin web and desktop UI packages consuming package-root APIs

Responsibilities by layer:

- Rust host: install, upgrade, supervise, migrate, back up, and bridge protocols
- API Router: provider routing, credentials, health, extension runtime, and loopback HTTP APIs
- external runtimes: upstream CLI, gateway, or agent behaviors
- frontend packages: UI, not runtime ownership

## Upgrade And Sync Strategy

Use a channel-aware sync pipeline instead of ad hoc manual updates.

Recommended flow:

1. track upstream tags, releases, installer changes, and migration notes
2. refresh the vendored analysis clones or pinned metadata
3. build a compatibility matrix per upstream version
4. verify install, `--version`, healthcheck, and one minimal functional action
5. update Hub Installer registry defaults only after verification

Recommended compatibility metadata per runtime:

- upstream repo URL
- pinned commit or release tag
- install provenance such as `source`, `release`, `script`, `global-user`
- config format version
- migration capability flags
- minimum supported version for in-product upgrades

## Backward Compatibility Strategy

- keep old registry names as compatibility aliases even when their implementation changes
- preserve `managed`, `partial`, and `opaque` control levels
- detect install provenance before upgrading
- keep one stable rollback target for every managed runtime
- run migrations only when capability checks confirm support
- back up config, data, and install records before every managed upgrade

## Immediate Corrections

The first corrective actions inside this repository should be:

- use official upstream URLs for ZeroClaw and IronClaw instead of stale fork references
- stop presenting ZeroClaw and IronClaw as `pnpm`-first tools in the bundled registry
- keep OpenClaw as the Node-managed exception
- make `sdkwork-claw-apirouter` and `sdkwork-claw-extensions` talk to real backends
- prioritize `API Router + Codex app-server + OpenClaw managed runtime` as the first end-to-end built-in integration

## Suggested Delivery Phases

## Phase 1

- correct registry metadata and install strategy defaults
- wire the real API Router backend into the frontend
- expose install provenance and runtime health in desktop UI

## Phase 2

- add Codex `app-server` process supervision and JSON-RPC bridge
- add managed OpenClaw process install plus lifecycle control
- persist runtime inventory and compatibility metadata

## Phase 3

- add ZeroClaw managed source or release integration
- add IronClaw managed source or release integration
- add import and migration UX for OpenClaw-derived data

## Phase 4

- add automated upstream sync jobs and compatibility verification
- add rollback and staged rollout flows for managed runtimes
- add upgrade advisories in the UI before destructive or incompatible transitions

## Notes

- This document is an integration strategy, not a guarantee that every upstream installer can be mirrored one-for-one inside the current Hub Installer manifest format.
- Where upstream packaging is inconsistent across platforms, prefer the most reliable managed path first and add platform-specific release fast paths only after verification.
